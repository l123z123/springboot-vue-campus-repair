# 校园报修管理系统 - 毕设论文技术资料

## 一、项目概述

### 1.1 项目名称
校园报修管理系统（Campus Repair Management System）

### 1.2 项目简介
基于 Vue3 + Spring Boot 的前后端分离校园报修管理系统（**派单版**业务与状态机以 `XQWD/` 与代码为准），面向学生、维修工、管理员三种角色，实现报修提交、**管理员审核派单、智能派单推荐**、工单流转、**邮箱验证码注册/登录（可配 QQ SMTP）**、WebSocket 实时通知、SLA 超时预警、数据统计、意见反馈等。

### 1.3 项目结构

## 二、系统架构

### 2.1 总体架构
- **前后端分离**：Vue3 单页应用 + Spring Boot REST API（`context-path=/api`）
- **通信方式**：HTTP（Axios）+ **WebSocket（STOMP，原生 `ws` + `@stomp/stompjs`，无 SockJS）** 广播工单与通知
- **认证方式**：**JWT**；Token 可写入 **Redis** 以便登出失效（Redis 不可用时自动降级为仅校验 JWT，与现网实现一致）
- **注册/登录增强**：**邮箱验证码**（发信走 **Spring Mail + SMTP**，如 QQ 邮箱）；验证码默认存 **Redis**，**Redis 不可用时** 邮箱模块可 **内存降级**（单机验收可完成流程，多实例需 Redis）
- **消息队列**：RabbitMQ 处理异步通知、延迟任务等（与具体消费者实现一致）
- **派单推荐**：管理员审核/派单时，系统基于维修工负载、同类经验、同区域经验和历史评分给出可解释推荐
- **定时任务**：**SLA 超时扫描**（如 `SlaTimeoutMonitor`）按配置小时数向管理员推送 `SLA_TIMEOUT`（payload 含 orderIds，管理端铃铛可点进详情）

### 4.3 安全与权限
- **公开访问（示例）**：`/auth/login`、`/auth/login-email`、`/auth/register`、`/auth/send-code`、`/auth/send-email-code`、`/auth/reset-password`、`/files/**`、**`/ws/**`**、Knife4j/Swagger 相关路径；具体以 `SecurityConfig` 为准。
- **管理员接口**（Controller 中 `/admin/**` 等）：需 **`ROLE_ADMIN`**（数字角色 2 映射为 `ROLE_ADMIN` 的实现以 `JwtUtil`/安全配置为准）。
- **其余业务接口**：需 **Bearer JWT**；过滤器为 `JwtAuthenticationFilter`。

### 4.4 工单状态与流转（派单版）

主线示例：**0 待审核 → 1 审核中 → 3 待派单 → 4 已派单 → 5 维修中 → 6 维修完成 → 7 学生确认 → 8 已评价**；另有 **9 已拒绝、10 已取消** 等终态。合法转移以 **`RepairOrderStatus` + `RepairServiceImpl`** 为准。旧文档中「0 待接单、1 进行中、2 已完成、4 已评价」的简化表述已废弃。

### 4.5 实时通知、智能派单与 SLA 预警
- **WebSocket**：`NotifyService` 经 `SimpMessagingTemplate` 向 **`/topic/notify`** 广播 JSON；`type` 如 `CREATED`、`STATUS_CHANGED`、`SLA_TIMEOUT`、`CHAT` 等。
- **智能派单推荐**：`RepairServiceImpl.recommendRepairmen` 根据维修工当前在办数量、同类故障处理经验、同区域处理经验和历史评价得分生成推荐列表与推荐理由，管理员最终确认派单。
- **SLA_TIMEOUT**：`SlaTimeoutMonitor` 按 `repair.sla.pending-hours`（默认 24 小时）扫描未闭环工单，payload 含 `count`、`orderIds`、message；仅管理员展示，管理端铃铛内对多单展示可点击工单号，跳转 `RepairDetail`。

### 6.2 启动步骤（与 `数据库_样例数据_README.txt` 一致时可更细）

1. `docker-compose up -d` 启动 **MySQL、Redis、RabbitMQ**（与 `docker-compose.yml` 一致）
2. **建库建表**：执行 `backend/src/main/resources/schema.sql`
3. **旧库若缺 `email` 列**：执行 `sql/migration_add_sys_user_email.sql`（新库若已含 `email` 可跳过）
4. **基础账号**：`backend/src/main/resources/init_data.sql`
5. **可选**：`init_all_accounts.sql`（多测试号）、`sql/fix_user_display_data.sql`（纠展示/手机号段）、`sql/seed_repair_orders_50.sql`（样例工单，可用 `python scripts/gen_seed_50.py` 重生成）
6. **测试数据清理**：若存在「**并发测试-**」标题工单，执行 `sql/delete_concurrent_test_repair_orders.sql`
7. `cd backend && mvn spring-boot:run` 启动后端（默认 `8081`，`context-path=/api`）
8. `cd frontend && npm run dev` 启动前端（Vite 将 `/api` 代理到后端）

## 八、论文可写技术要点

1. **前后端分离**：Vue3 + Vite + Spring Boot REST，职责清晰，便于维护。
2. **派单版工单状态机**：与校园后勤实际一致，采用管理员审核派单、维修工处理、学生确认评价的闭环状态机。
3. **JWT 与权限**：无状态 Token；可配合 Redis 做登出失效；Spring Security 区分公开接口与 `ROLE_ADMIN` 等。
4. **注册与真实身份验证**：**邮箱验证码**（Spring Mail + SMTP，QQ 邮箱授权码模式）、`sys_user.email` 唯一；支持 **邮箱验证码登录** 与 **密码登录** 并存。
5. **高并发与测试数据**：并发创建工单用例及 SQL 清理脚本、内存/Redis 策略（可简述）。
6. **消息队列与异步**：RabbitMQ 在通知、延迟等路径中的作用（与现网消费者一致处着笔）。
7. **WebSocket 与 SLA 预警**：STOMP 经 `/topic/notify` 推送；SLA 超时预警携带 orderIds，管理端可定位具体工单。
8. **智能派单推荐**：基于维修工负载、同类经验、区域经验和历史评价的可解释规则评分，辅助管理员派单。
9. **数据可视化**：ECharts 展示区域、状态、类型、时段、维修工绩效等。
10. **可观测与降级**：Redis 不可用时部分能力降级、全局异常 503/业务错误分层（可引用 `GlobalExceptionHandler` 设计）。
11. **容器与本地部署**：Docker Compose 启动 MySQL/Redis/RabbitMQ 及端口映射，与 `application.yml` 对齐。
