# Superpowers 使用指南

> Superpowers 是一套安装在 Claude Code 里的"技能包"（Skills）。每个 Skill 是一个专门的工作流模板，擅长处理特定类型的任务。可以理解为 Claude Code 原生能力之上的专业工具箱。

## 一、开发流程类

### `/brainstorming`
**一句话**：动手前先想清楚需求和方案。

**什么时候用**：加新功能、改行为、做任何创意性工作前。避免直接写代码导致方向跑偏。

### `/writing-plans`
**一句话**：把需求变成可执行的步骤计划。

**什么时候用**：拿到明确需求后，动手写代码之前。会产出 `docs/superpowers/plans/` 目录下的计划文件。

### `/executing-plans`
**一句话**：按计划逐步执行，每步有 review checkpoint。

**什么时候用**：有了 plan 之后，在独立 session 里执行。适合长任务，不会因为对话中断丢失进度。

### `/test-driven-development`
**一句话**：先写测试再写实现。

**什么时候用**：修 bug 或加新功能时。强制测试先行，减少回归问题。

### `/verification-before-completion`
**一句话**：完工前跑验证，确认真的没问题。

**什么时候用**：准备说"做完了"之前。要求运行验证命令并确认输出，然后才能声称成功。

## 二、代码审查类

### `/review-changes`
**一句话**：结构化审查所有改动。

**什么时候用**：完成一个功能后想自查。使用变更检测和影响分析。

### `/review-delta`
**一句话**：只看上次 commit 之后的改动。

**什么时候用**：日常提交前快速检查。Token 高效的增量审查，自动检测影响范围。

### `/review-pr`
**一句话**：审查 PR 差异，含知识图谱结构分析。

**什么时候用**：合并 PR 前。输出结构化审查报告，含影响范围分析。

### `/requesting-code-review`
**一句话**：准备把你的代码提交审查。

**什么时候用**：功能完成、准备合并时。验证工作满足要求再提交。

### `/receiving-code-review`
**一句话**：收到审查意见后理性分析再改。

**什么时候用**：别人给你提了 review 意见。要求技术严谨性验证，而非盲目接受或敷衍同意。

### `/security-review`
**一句话**：安全审查当前分支的改动。

**什么时候用**：涉及权限、认证、敏感数据、JWT、Redis 等安全相关改动时。

---

## 三、调试重构类

### `/systematic-debugging`
**一句话**：系统化排查 bug，不做猜测性修复。

**什么时候用**：遇到任何 bug 或测试失败。使用知识图谱导航代码来定位根因。

### `/debug-issue`
**一句话**：用知识图谱导航代码来定位问题。

**什么时候用**：复杂 bug，不确定问题出在哪个文件。

### `/refactor-safely`
**一句话**：分析依赖关系后安全重构。

**什么时候用**：想改架构、抽公共方法、调整模块划分时。基于依赖分析制定重构计划。

### `/simplify`
**一句话**：检查改动代码的复用性、质量和效率。

**什么时候用**：写完代码后想优化。审查改动并修复发现的问题。

## 四、项目管理类

### `/using-git-worktrees`
**一句话**：创建隔离的工作区，不影响当前分支。

**什么时候用**：开始一个大功能，不想搞乱当前代码。或执行实施计划前需要隔离环境。

### `/finishing-a-development-branch`
**一句话**：开发完成后的收尾。

**什么时候用**：功能做完、测试通过后。提供合并、PR、清理等结构化选项。

### `/dispatching-parallel-agents`
**一句话**：同时派多个 Agent 干独立任务。

**什么时候用**：有 2 个以上互不依赖的任务时，并行执行节省时间。

### `/huashu-agent-swarm`
**一句话**：蜂群模式，多 Agent 纯 git 自组织协作。

**什么时候用**：大型项目需要多人并行开发。适合复杂的大型项目。

## 五、文档文件处理类

### `/docx`
**一句话**：Word 文档的创建、编辑、读取。

**什么时候用**：操作 .docx 文件。包括内容提取、格式调整、图片替换、批注处理等。

### `/pdf`
**一句话**：PDF 的读取、合并、拆分、加水印。

**什么时候用**：操作 .pdf 文件。包括文本提取、合并拆分、加密解密、OCR 等。

### `/pptx`
**一句话**：PPT 的创建和编辑。

**什么时候用**：做演示文稿、幻灯片。包括模板应用、布局调整、备注编辑等。

### `/xlsx`
**一句话**：Excel 电子表格处理。

**什么时候用**：操作 Excel 文件。包括公式计算、格式化、图表生成、数据清洗等。

### `/huashu-md-to-pdf`
**一句话**：Markdown 转精美 PDF 白皮书（苹果设计风格）。

**什么时候用**：技术文档、白皮书、教程等需要专业排版的 Markdown 文档导出。

### `/huashu-slides`
**一句话**：从内容到成品 PPTX 端到端制作。

**什么时候用**：做 PPT 演讲。含 AI 插画生成和 18 种设计风格。

## 六、内容创作类

### `/huashu-article-edit`
**一句话**：标准化文章编辑流程。

**什么时候用**：编辑/修改文章时。确保修改范围明确、进度可追踪、变更有记录。

### `/huashu-proofreading`
**一句话**：三遍审校降低 AI 检测率，让文章更有人味。

**什么时候用**：感觉文章"AI 味太重"时。

### `/humanizer-zh`
**一句话**：去除中文文本 AI 生成痕迹。

**什么时候用**：让论文/文章读起来更自然。基于维基百科"AI 写作特征"指南，检测并修复：夸大象征意义、宣传性语言、肤浅分析、模糊归因、破折号过度使用等。

### `/huashu-research`
**一句话**：结构化网络调研，结果保存到文件。

**什么时候用**：需要查资料、调研某个话题。增量保存，不会因会话截断丢失。

### `/huashu-topic-gen`
**一句话**：快速生成 3-4 个选题方向，含标题、大纲和优劣分析。

**什么时候用**：不知道写什么文章时。

### `/huashu-speech-coach`
**一句话**：演讲与分享教练。

**什么时候用**：准备答辩、技术分享、培训讲课。基于 MIT Patrick Winston 的 How to Speak 方法论。

### `/huashu-article-to-x`
**一句话**：长文精简为社交媒体内容（200-500字）。

**什么时候用**：把文章转成微博、小红书等内容，保留核心观点和个人风格。

## 七、UI前端设计类

### `/frontend-design`
**一句话**：生产级前端界面，避免 AI 通用风格。

**什么时候用**：做网页、组件、Landing Page、Dashboard 等。生成有创意的、高质量的前端代码。

### `/ui-ux-pro-max`
**一句话**：UI/UX 设计智能库。

**什么时候用**：设计评审、选配色、改 UI。包含 50+ 风格、161 调色板、57 字体配对、99 UX 指南、25 图表类型。

### `/huashu-design`
**一句话**：设计哲学顾问。

**什么时候用**：不知道用什么设计风格时。从 20 种风格中推荐 3 个方向并生成视觉 Demo。

## 八、运营自媒体类

### `/huashu-douyin-script`
**一句话**：抖音爆款脚本创作全流程。

**什么时候用**：做抖音视频。从竞品视频拆解到脚本生成的完整流程。

### `/huashu-video-outline`
**一句话**：快速生成 2-3 个视频大纲方案。

**什么时候用**：视频选题策划。含标题、封面建议和结构设计。

### `/huashu-video-check`
**一句话**：基于 MrBeast 策略检查视频标题、封面和开头钩子。

**什么时候用**：优化视频点击率和观看时长。

### `/huashu-wechat-image`
**一句话**：公众号配图生成（封面 2.35:1、正文插图 16:9/4:3、信息图）。

**什么时候用**：写公众号文章需要配图时。

### `/huashu-xhs-image`
**一句话**：小红书笔记配图生成。

**什么时候用**：做小红书笔记需要配图时。默认 AI 生成。

## 九、系统配置工具类

### `/update-config`
**一句话**：修改 Claude Code 的 settings.json。

**什么时候用**：改权限、加环境变量、配置 hook、调整自动化行为。

### `/find-skills`
**一句话**：发现和安装新的 skill。

**什么时候用**：想找有没有某个功能的 skill 可以安装。

### `/skill-creator`
**一句话**：创建或修改 skill。

**什么时候用**：想做自己的定制化 skill，或优化现有 skill。

### `/init`
**一句话**：初始化项目的 CLAUDE.md 文件。

**什么时候用**：新项目需要写项目文档，让 Claude 理解代码库。

### `/build-graph`
**一句话**：构建或更新代码知识图谱。

**什么时候用**：首次使用审查/探索功能前。或代码结构有重大变化后。

### `/fewer-permission-prompts`
**一句话**：减少权限弹窗。

**什么时候用**：觉得每次都要点"允许"太烦。扫描常用只读命令自动添加到 allowlist。

### `/loop`
**一句话**：定时循环执行命令或 slash command。

**什么时候用**：需要轮询检查某个状态，比如每 5 分钟检查一次部署状态。

## 典型工作流

### 完整开发流程

```
/brainstorming        → 想清楚要做什么
        ↓
/writing-plans        → 产出执行计划
        ↓
/executing-plans      → 按计划逐步实现
        ↓
/test-driven-development  → 保证质量
        ↓
/review-changes       → 自查代码
        ↓
/verification-before-completion  → 最终确认
        ↓
/finishing-a-development-branch  → 合并收尾
```

### 论文写作流程

```
/writing-plans          → 制定写作计划
        ↓
/huashu-research        → 调研参考文献
        ↓
（按 plan 逐章写作）
        ↓
/humanizer-zh           → 去 AI 味
        ↓
/huashu-md-to-pdf       → 导出精美 PDF
```

### 答辩准备流程

```
/huashu-slides          → 制作答辩 PPT
        ↓
/huashu-speech-coach    → 准备演讲内容
        ↓
（排练）
```

---

> **提示**：不确定用哪个 Skill 时，直接用自然语言描述你想做什么，Claude 会自动匹配合适的 Skill。
