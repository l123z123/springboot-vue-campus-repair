# 第2章 相关技术介绍

本章将对系统开发所依赖的核心技术方案逐一进行说明，使后续设计章节的讨论有据可依。

## 2.1 开发环境

本系统在Windows 11平台上完成开发与部署，后端代码编写选用IntelliJ IDEA，前端界面开发则使用Visual Studio Code。依赖管理和项目构建方面，后端借助Maven统一管理JAR包依赖，前端以Vite作为模块打包器并提供开发阶段的即时热更新。版本管理工具为Git，远程仓库托管在GitHub。数据库与Redis缓存等基础服务全部容器化到Docker Compose中统一编排，以此确保开发与生产环境的高度一致[1]。

## 2.2 MySQL 8.0

相较前代版本，MySQL 8.0在多个维度均有显著改进，包括查询执行效率提升、新增窗口函数与公共表表达式（CTE）的支持，以及对JSON数据类型的原生处理能力[2]。

数据存储层选用MySQL 8.0.36，以Docker容器方式运行并映射到3307端口，数据库实例命名为campus_repair，默认字符集设为utf8mb4以确保emoji等扩展Unicode字符的兼容性。持久层框架采用MyBatis-Plus，借助LambdaQueryWrapper以类型安全的方式构造查询条件，从根本上规避了字符串硬拼接带来的SQL注入隐患。针对工单列表等高频分页场景，在user_id、repairman_id、status等查询字段上建立B+树索引以保证响应速度[3]。

## 2.3 Redis 6.2

自6.2版本起，Redis新增了访问控制列表（ACL）功能，使得在多用户运维场景下的权限隔离与安全管理更加完善[4]。

缓存层采用Redis 6.2.14，同样以Docker容器部署在6379端口并启用密码保护。客户端连接器选用Jedis而非Lettuce，原因在于Jedis提供的同步API更为简洁，与校园管理系统这类中等并发量场景十分匹配。Redis承担两项缓存职责：其一为JWT令牌的缓存，采用`auth:token:{userId}`键结构存储，使得用户登出时可由服务端主动销毁令牌；其二为邮箱验证码的临时缓存，键结构为`auth:email:code:{scene}:{email}`，每条验证码5分钟后自动过期，且同一邮箱在60秒窗口内禁止重复触发[5]。

为应对Redis宕机等异常情况，系统内置了自动降级机制。一旦检测到Redis不可达，JWT的签发流程不受影响，仅丧失服务端主动吊销令牌的能力；验证码存储则自动回退至JVM进程内的ConcurrentHashMap，使注册与登录功能依然可用[6]。

## 2.4 后端框架

### 2.4.1 Spring Boot 2.7

Spring Boot是在Spring Framework之上构建的简化框架，其核心价值在于自动配置——开发者无需编写大量XML配置文件即可快速启动项目[7]。

后端服务建立在Spring Boot 2.7.18之上，运行于Java 8环境。框架的自动配置能力贯穿数据源、事务管理器和Web MVC等关键组件。代码组织遵循经典的三层架构：Controller层负责接收请求并执行入参校验，Service层承载核心业务逻辑与事务边界，Mapper层基于MyBatis-Plus完成数据库交互。所有未捕获的异常由GlobalExceptionHandler集中拦截，统一包装为规范的JSON错误体返回给客户端[8]。

### 2.4.2 MyBatis-Plus 3.5

MyBatis-Plus是对MyBatis的扩展增强，完全兼容原生MyBatis的同时，额外提供了通用Mapper、条件构造器、分页插件、逻辑删除和乐观锁等一系列开箱即用的能力，显著降低了重复性CRUD编码量[9]。

本系统集成的是MyBatis-Plus 3.5.5。实体映射方面，@TableName将Java类与数据库表关联，@TableId配合雪花算法（IdType.ASSIGN_ID）为每条记录生成全局唯一主键。分页功能依赖Page对象与LambdaQueryWrapper的链式组合，可按状态、区域、类型等多个维度灵活筛选。并发控制方面，利用version字段加@Version注解实现乐观锁，有效防止管理员派单与维修工接单同时操作同一条工单时的数据冲突[10]。

### 2.4.3 Spring Security与JWT认证

Spring Security作为Spring生态的安全子框架，核心职能涵盖认证（Authentication）与授权（Authorization）两个层面。SecurityConfig类集中定义了安全策略，将系统接口划分为公开与受保护两类：登录、注册、验证码发送等认证接口，以及/files/**文件上传路径、/ws/** WebSocket握手路径和Knife4j接口文档路径均被放行；除此之外的所有业务接口必须附带合法的Bearer JWT令牌方可访问[1]。

JWT（JSON Web Token）是一种轻量、URL安全的自包含令牌，结构上由Header、Payload和Signature三段Base64编码拼接而成。本系统选用jjwt 0.11.5库完成令牌的签发、解析与合法性验证。Payload段内嵌用户ID、用户名和角色信息，整体有效期为2小时。在每次请求到达时，JwtAuthenticationFilter自动从Authorization头提取Bearer Token并完成校验，随后将解析出的用户身份写入SecurityContextHolder供后续处理链使用。结合Spring Security的@PreAuthorize注解及路径匹配规则，最终实现细粒度的方法级角色访问控制[2]。

### 2.4.4 Spring Mail邮件服务

Spring Mail是对底层JavaMail API的高层封装，通过MailSender与JavaMailSender两大接口降低了邮件编程的复杂度。系统引入spring-boot-starter-mail启动器后，在application.yml中完成SMTP服务器地址、发件邮箱账号及授权码的配置。实际发送环节使用SimpleMailMessage构造纯文本邮件体，覆盖邮箱验证码下发和系统通知两个场景[3]。

## 2.5 前端框架

### 2.5.1 Vue 3

Vue 3的革新闻之一在于引入了组合式API（Composition API），使逻辑复用和代码组织更加灵活；底层响应式机制也从Object.defineProperty升级为ES6 Proxy，消除了旧版本在数组监听和动态属性追踪上的若干限制[4]。

前端基于Vue 3.4.0搭建，构建工具选用Vite 5.0。Vite利用浏览器原生ES模块能力实现开发服务器的快速冷启动和热模块替换（HMR），构建效率远超Webpack等传统打包器。路由层采用Vue Router 4.2.5，根据用户身份动态注册不同角色的路由表。全局状态管理放弃传统的Vuex，改用Pinia 2.1.7——后者API更精简、TypeScript支持更友好。系统定义了user store（管理令牌与用户资料）和systemNotify store（维护管理端WebSocket通知队列）两个状态容器[5]。

### 2.5.2 Element Plus 2.4

Element Plus是专为Vue 3设计的桌面端组件库，覆盖按钮、表格、表单、对话框、导航菜单等常见UI元素。管理端和维修工端的所有后台界面均基于该库搭建，保证了视觉风格的统一。表单验证交给Form组件的内置规则引擎处理；表格的分页、排序和筛选均通过Table组件的声明式属性配置完成；对于需要自定义渲染的单元格，则可借助Slot插槽机制灵活实现[6]。

### 2.5.3 ECharts 6.0

ECharts是Apache基金会旗下的开源可视化库，内置折线图、柱状图、饼图、散点图、热力图、旭日图等数十种图表形态。6.0版本重点优化了大数据集下的渲染吞吐率，并改进了移动终端上的手势交互体验[7]。

在管理端数据分析面板中，系统借助ECharts 6.0从多个维度对运维数据进行可视化：区域报修Top10以柱状图揭示报修在空间上的聚集特征；工单状态分布以饼图呈现系统当前运行的全貌；故障类型层级关系以旭日图表达，支持从一级分类钻取至子分类；报修时段分布以热力图展示，通过颜色梯度区分各时段的活跃度高低；维修工绩效则以横向柱状图并列对比每个人的完工数量和平均评分[8]。

## 2.6 WebSocket与STOMP协议

WebSocket是HTML5标准引入的全双工通信协议，在单一TCP连接上即可维持客户端与服务端的双向实时数据通道，彻底消除了传统HTTP轮询模式下持续建立连接所造成的带宽浪费与延迟积压[1]。

STOMP（Simple Text Oriented Messaging Protocol）是运行在WebSocket之上的文本消息子协议，规范了消息的发送、订阅与路由语义。Spring通过spring-websocket模块原生集成了STOMP协议栈。本系统在WebSocketConfig配置类中激活STOMP消息代理，将`/topic`前缀作为内置Broker的广播地址空间，客户端只需订阅`/topic/notify`主题即可实时接收服务端推送的通知。服务端一侧，调用SimpMessagingTemplate的convertAndSend方法即可向指定目标投递JSON格式的消息体[2]。

本系统中，WebSocket的实时推送能力主要服务于两类通知场景。第一类是工单状态流转通知：每当工单在其状态机中发生迁移（例如学生提交报修、管理员审核放行、维修工标记完工），NotifyService即向管理端铃铛组件推送CREATED或STATUS_CHANGED类型消息。第二类是SLA超时预警通知：SlaTimeoutMonitor定时任务周期性扫描超时工单，发现积压情况后向管理员广播SLA_TIMEOUT类型消息，消息体中携带超时工单总数及对应的工单ID列表，管理员点击通知条目即可直接跳转到对应工单的详情页[3]。
