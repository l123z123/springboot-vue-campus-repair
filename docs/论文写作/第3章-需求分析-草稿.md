# 第3章 系统需求分析

围绕校园报修管理系统，本章依次从可行性论证、总体架构、功能划分、业务流程、角色用例及非功能性约束六个维度展开需求分析，以此作为后续设计开发的基础参照。

## 3.1 可行性分析

**技术可行性。** 在技术选型上，后端以Spring Boot 2.7为核心框架，利用其内嵌Tomcat与起步依赖机制，将传统Java EE项目中的容器配置与依赖管理环节大幅简化。数据访问层选用MyBatis-Plus 3.5——该框架在MyBatis基础上扩展了通用CRUD封装、分页插件与Lambda条件构造器，使开发者得以显著减少XML映射文件的编写量。前端方面，Vue 3的组合式API配合Element Plus组件库及Vite构建链，支撑起组件化与模块化的开发范式。数据存储层面，MySQL 8.0承担事务型业务数据的持久化并提供完整的ACID保障，Redis 6.2则以内存级缓存应对高并发读取场景。上述各项技术均有成熟的社区生态与大量线上案例验证，整体技术风险处于可控区间[1][2]。

**经济可行性。** 系统所依赖的软件栈全部来自开源社区，不存在商业许可费用：Spring Boot、MyBatis-Plus、MySQL Community Edition、Redis（开源发行版）、Vue 3及Element Plus均免费可用。IDE工具方面，后端开发使用IntelliJ IDEA社区版（无需付费），前端开发使用VS Code（同样免费）。在部署环节，依托Docker Compose容器化编排方案，系统可以运行于普通PC或常规云服务器之上，不需要采购专用硬件设备。综合来看，研发与运维成本均在可控范围之内，经济层面具备充分的可行性[3]。

**操作可行性。** 系统沿用了成熟的B/S架构模式——用户端零安装，打开浏览器即可访问全部功能。面向学生与维修工的前端界面采用响应式设计，能够同时适配个人电脑与移动端浏览器的不同视口尺寸。管理员侧则提供PC端专用管理后台，配备功能导航体系和批量操作入口。三种角色的操作流程均按照高校后勤报修的真实业务场景设计，使用者经过简短介绍后即可独立完成操作，上手成本很低[4]。

## 3.2 系统总体架构

系统采用前后端解耦的B/S分层设计，自顶向下划分为展示层、服务层与持久层。

（1）**前端展示层**：以Vue 3为基底构建单页应用（SPA），路由由Vue Router统一托管，全局状态交由Pinia集中维护，HTTP通信通过Axios统一封装，WebSocket实时通道则借助@stomp/stompjs实现。页面组件按照角色边界拆分为学生端页面组、维修工端页面组与管理员端页面组，三者共用底层的API服务模块及公共UI组件。

（2）**后端服务层**：以Spring Boot为骨架搭建RESTful API服务体系，内部遵循Controller-Service-Mapper的分层约定——Controller负责接收HTTP请求并完成参数校验，Service承载核心业务逻辑，Mapper则通过MyBatis-Plus完成数据库交互。认证鉴权由Spring Security接管，自定义的JwtAuthenticationFilter在请求链路中拦截并解析JWT令牌以完成身份识别。实时通知模块借助WebSocket+STOMP协议将消息推送至客户端。SLA超时监控以独立定时任务的形式在Spring Scheduler线程池中运行。

（3）**数据持久层**：MySQL 8.0作为关系型主库，负责存储用户、工单、评价等核心业务记录。Redis 6.2充当缓存中间层，承载JWT令牌与邮箱验证码的临时存储；当Redis连接异常时会自动切换为进程内存模式以保证服务不中断。

三种角色——学生、维修工、管理员——各自拥有独立的操作界面与权限边界：学生通过响应式页面发起报修并追踪处理进度；维修工在工作台中查看已分配工单并执行维修任务；管理员则通过PC管理后台完成审核派单、用户管理及数据统计分析等工作[5]。

## 3.3 功能需求分析

基于对高校后勤报修业务的实地调研，并参照山锋[3]所总结的传统报修模式五大痛点——报修渠道单一导致信息展示不充分、报修数据精度不足难以快速定位、维修进度对报修人不可见缺乏监督、维修工工作量无法实时量化、统计汇总环节繁琐低效——本系统面向三类角色分别进行了功能模块的规划。

**学生端功能。** 学生兼具报修发起者与服务质量评价者的双重身份。围绕这一身份，系统为其提供以下功能：账号注册与登录（支持密码认证与邮箱验证码认证两种方式）、报修提交（填写标题、故障描述、地点、联系电话、校区归属、区域类型、故障类别、紧急等级，并可上传现场实拍图片）、工单进度追踪（查看已提交工单的处理状态与完整的流转时间线）、在线沟通（在工单详情页内与承接维修工进行即时文字交流以补充故障信息）、完工确认（对维修工标记为已完成的任务进行核实确认）、服务评价（从维修质量、响应效率、服务态度三个维度进行星级评分并撰写文字评价）。

**维修工端功能。** 维修工作为工单的实际执行者，其功能围绕任务接收与处理链条展开：工作台看板（以统计卡片形式汇总展示各状态下的工单数量，如待处理、维修进行中、等待确认等）、工单列表（支持按状态筛选并查看分配至本人的工单集合）、接单与开工（将状态为"已派单"的工单流转为"维修中"）、完工上报（维修结束后上传完工照片与维修说明，推动工单状态进入"维修完成"）、历史工单（回溯查询本人处理过的全部工单记录）。

**管理员端功能。** 管理员定位为系统的全局管控者与调度决策者，其功能模块设计如下：数据仪表盘（以指标卡片概览当日报修量、累计工单总量、待处理数量、已完成数量、紧急工单数等关键数据）、工单管理（浏览全量工单列表，可按状态、区域、故障类型等多个维度组合筛选）、审核与派单（对待审核工单进行真实性核验，核验通过后执行派单操作，系统内置推荐排序算法为派单决策提供参考依据）、用户管理（对三类角色账号进行增删改查，支持按角色与院系维度过滤，可对账号执行启用/停用操作）、数据统计（通过ECharts可视化图表呈现区域报修Top10排行、工单状态分布比例、故障类型分类树、时段报修热力图、维修工绩效排行榜等多维分析视图）、反馈管理（查阅用户提交的意见反馈并标记当前处理状态）。上述功能模块的划分方式参考了肖智兵论文[5]中按角色划分子系统的思路。

## 3.4 业务流程分析

系统所承载的业务流程构成了一条从报修发起到服务评价的完整闭环链路，具体分为以下四个阶段[6]：

（1）**报修提交阶段**：学生完成登录后进入报修填报界面，依次输入标题、详细描述、地点、联系电话，选定校区（河东/河西）、区域分类（宿舍/公寓/教学楼等）、故障类别（水电/门窗/空调/多媒体设备等）与紧急等级（低/中/高/一键急诊），并上传反映现场情况的图片。提交成功后，该工单自动进入"待审核"状态。

（2）**审核派单阶段**：管理员在工单管理视图中定位待审核工单，对报修信息的真实性与完整性进行审查。审核予以通过后，系统触发推荐算法——该算法综合考量维修工当前的待处理负载、同类故障的处置经验、同区域服务经验以及历史评分四个维度——生成按推荐度降序排列的维修工候选列表。管理员参考此排序结果选定执行人并完成派单操作，工单随之进入"已派单"状态并与指定维修工建立绑定关系。

（3）**维修处理阶段**：维修工从工作台或工单列表视图中获取已分配至本人的工单，确认接收后启动维修工作，工单状态相应地置为"维修中"。在维修进行期间，学生与维修工可借助工单内嵌的即时沟通功能就故障细节进行交流。维修作业完毕后，维修工上传完工现场照片并撰写维修说明，提交后状态流转至"维修完成"。

（4）**确认评价阶段**：学生端展示维修完成信息及完工照片，学生核实无误后执行确认操作，状态更新为"学生确认"。随后学生可从评分（1-5星）与文字评价两个层面对本次维修服务做出反馈，提交完成后状态最终变为"已评价"，至此工单达成终态、流程闭环[7]。

此外，系统内置了SLA超时预警机制：若某工单在"待审核""审核中"或"待派单"任一状态下滞留超过24小时而未被处理，后台定时任务将自动取消该工单并通过WebSocket向管理员端推送预警通知[8]。

## 3.5 角色用例分析

系统包含三种用户角色，以下分别描述各角色的用例集合：

**学生用例。** 学生角色的核心用例涵盖：账号注册、系统登录、提交报修申请、浏览个人报修列表、查看单个工单的详情及状态变更轨迹、在工单内与维修工进行信息沟通、对已完成维修的工单进行确认验收并提交评价。学生侧的操作均承载于学生端与维修工端共享的响应式界面上，通过移动端的底部导航栏或PC端的侧边导航栏在各功能页面之间切换。

**维修工用例。** 维修工角色的核心用例涵盖：系统登录、查看工作台统计概览、浏览分配至个人的工单列表、接单并启动维修、维修期间与报修学生进行线上交流、完工后上传完工信息、回溯查询个人历史工单。需要指出的是，维修工无权查看不属于自身的其他工单详情，也不具备派单决策与用户管理的操作权限。

**管理员用例。** 管理员角色的核心用例涵盖：系统登录、查看数据仪表盘、对全部工单执行管理操作（浏览、筛选、审核、派单）、参考派单推荐信息、对用户账号进行管理（查询、新建、编辑、禁用）、查阅数据统计图表、处理用户反馈、接收系统实时消息与SLA超时预警通知。管理员拥有系统内的最高操作权限，其所有关键操作均由AOP操作日志模块进行全程记录。

## 3.6 非功能性需求

**性能需求。** 系统的并发承载目标设定为同时支持至少50名用户在线操作，常规页面应在3秒内完成加载渲染。为达成这一指标，数据库层面为高频查询字段（用户ID、工单状态、创建时间）建立了索引以加速检索；缓存层通过Redis缓存热点查询结果来削减数据库的直接负载压力；前端则借助路由懒加载与组件按需引入策略来压缩首屏的资源请求体积。

**安全性需求。** 用户密码采用BCrypt算法进行不可逆加密后存入数据库，杜绝明文泄露风险。身份认证基于JWT令牌体系实现——令牌设定2小时有效期，用户登出时由Redis记录失效令牌以阻断后续非法使用。接口层面的权限控制委托给Spring Security，按角色粒度进行拦截：所有以/admin/**为前缀的管理员专属接口仅对ROLE_ADMIN角色开放，学生与维修工无法实施越权访问。对于登录、注册、状态变更等敏感操作，统一经由AOP切面记录审计日志，日志内容涵盖操作人ID、操作类型、来源IP、请求参数及操作时间戳。

**可用性需求。** 系统的可用性设计围绕Redis降级策略展开：一旦Redis服务出现不可用状况，JWT令牌的签发与校验流程不受影响（回退为仅依赖JWT自身签名进行验证），邮箱验证码的存储则自动切换至进程内的ConcurrentHashMap容器中。前端统一捕获网络异常与各类HTTP错误码（如401未登录、403权限不足、500服务器内部错误等），向用户呈现经过语义转换的可理解提示文字，而非原始的技术异常堆栈信息。

**可维护性需求。** 后端遵循Controller-Service-Mapper的经典分层结构，各层职责边界清晰、耦合可控。数据库主键采用雪花算法统一生成，确保在分布式部署情景下的全局唯一性，消除主键冲突隐患。前端代码按功能模块组织目录层次，所有API请求集中在api/目录下统一管理，全局状态由Pinia store进行集中维护——这些设计都能为后续的迭代开发与系统维护提供便利。
