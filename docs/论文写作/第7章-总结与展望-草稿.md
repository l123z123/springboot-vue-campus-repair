# 第7章 总结与展望

## 7.1 工作总结

本文从高校后勤报修业务的现实痛点切入，完成了一套基于SpringBoot+Vue前后端分离模式的校园报修管理系统的设计与开发。全文沿软件工程的标准生命周期推进，依次覆盖了需求分析、技术选型、系统设计、功能实现和系统测试五个阶段，取得的主要成果归纳如下：

（1）**系统化的需求分析**。在调研高校后勤报修实际业务流程的基础上，综合现有文献的研究结论，归纳出传统报修模式存在的五项核心缺陷——报修渠道单一、数据记录不精确、进度不可追踪、维修工缺乏监督机制、统计汇总工作繁琐——进而围绕学生、维修工和管理员三类用户角色构建了完整的功能需求体系。

（2）**清晰的系统架构设计**。采用前后端分离的B/S三层模型：前端依托Vue 3组合式API与Element Plus组件库搭建响应式界面；后端以Spring Boot 2.7为底座提供RESTful API服务；MySQL 8.0承担持久化存储任务，Redis 6.2负责缓存加速。系统按功能拆解为认证、工单、通知、统计、用户、反馈、沟通七个模块，各模块之间通过标准化接口松耦合交互。

（3）**核心功能的完整实现**。用户认证层融合Spring Security与JWT，同时支持密码和邮箱验证码两种登录形式。工单流转采用包含过渡态的11状态状态机模型，完整覆盖从报修提交到评价闭环的全链路。智能派单推荐从负载、同类经验、同区经验和历史评分四个维度加权打分，为管理员派单决策提供可量化的数据参考。消息推送基于STOMP over WebSocket实现实时通信，规避了轮询模式的延迟和带宽开销。SLA超时预警以定时任务自动扫描积压工单并触发告警。ECharts多维统计图表则将后勤运维数据转化为直观的决策依据。

（4）**系统测试**。设计并执行了11项功能测试用例，涉及认证、工单、派单推荐、通知、统计和并发六大模块，全部测试用例一次通过。测试阶段还定位并修复了三项关键技术缺陷——Redis认证兼容性问题、LocalDateTime序列化异常、雪花ID在前端的精度溢出——有力保障了系统的运行稳定性。

## 7.2 不足与展望

尽管系统已具备完整可用性并通过功能测试，但受限于开发周期和个人技术水平，以下方面仍有改进余地：

（1）**推荐算法的精细化**。当前推荐引擎采用固定权重方案（负载因子-20、同类经验加8上限5、同区经验加5上限5、历史评分乘4），各项系数均来自主观经验，未经过历史数据的回归校准。后续可积累真实派单记录作为训练样本，借助机器学习自动拟合最优权重，提高推荐命中率。

（2）**移动端原生应用缺失**。当前前端仅以响应式Web形态适配手机浏览器，在推送触达的实时性、离线场景的可用性以及与系统级通知的无缝整合上，均与原生App存在明显差距。后续可基于现有RESTful API开发独立的原生移动客户端，或采用uniapp等跨平台方案以较低成本覆盖iOS和Android。

（3）**系统监控与运维能力的增强**。当前运维仅停留在GlobalExceptionHandler统一记录异常日志的层面，缺乏对系统运行状态的主动监测和预警。后续可引入Spring Boot Actuator暴露运行指标，搭配Prometheus收集数据、Grafana展示面板，对JVM堆内存使用率、API响应延迟、数据库连接池饱和度等关键指标建立实时监控和阈值告警体系。

（4）**消息中间件的引入**。当前消息推送采用WebSocket直连模式，单机部署时完全满足需求，但如果未来扩展为多节点集群，WebSocket会话会散布在不同节点上，跨节点广播将变得困难。后续可引入RabbitMQ或RocketMQ等消息队列作为中间层，将服务端广播消息先发布至队列、再由各节点消费后通过WebSocket下发，从而在集群环境下实现可靠的消息送达。

（5）**审批流程的灵活化**。目前审核与派单的流程是固定的——所有工单统一由管理员审核通过后再人工指派。后续可引入可配置的审批策略引擎，针对不同紧急等级或故障类别设置差异化路径：例如紧急工单自动跳过审核直达派单，小额维修允许维修工自行认领，以进一步提升流转效率。

（6）**数据安全性的强化**。现有安全手段主要包括BCrypt密码哈希存储和JWT令牌认证，但在通信链路加密（全站HTTPS）、敏感字段的展示脱敏、操作审计日志的全覆盖等方面仍显不足。后续可依据等保2.0标准体系，分步补齐上述安全短板。

校园报修管理系统是智慧校园体系中的一环，对提高后勤管理效率、优化师生报修体验具有切实的落地价值。伴随技术演进和业务需求的变化，本系统将持续迭代完善，为高校后勤信息化与智能化建设贡献力量。
