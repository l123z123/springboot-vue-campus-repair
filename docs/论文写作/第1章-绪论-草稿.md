# 第1章 绪论

## 1.1 系统开发背景

教育信息化是国家教育现代化的关键支撑。《中国教育现代化2035》明确指出应加速信息化背景下的教育转型，推进智能化校园建设，构建教学、管理与服务融合的一体化智能平台[1]。智慧校园既是高校管理效能提升的抓手，也属于教育信息化国家战略的重要组成部分。

后勤报修管理在高校日常运转中占据重要位置，其运转效率与师生群体的学习体验和生活便利度紧密关联。但从实际情况看，相当一部分高校仍依赖电话、纸质表格或简易Excel记录来管理报修事务，暴露出一系列不足[2]：

（1）**报修渠道单一，信息展示不足**：学生仅能通过拨打电话或亲赴后勤办公点进行报修登记，处理进展无法在线查看，整个过程缺少必要的信息透明度。

（2）**报修信息模糊，定位困难**：口头转述或文字登记难以准确描述故障点位和实际状况，维修人员往往需要往返确认，沟通与时间开销随之增加。

（3）**维修进度缺少追踪手段**：报修一旦提交，学生对后续节点——是否已分配人员、是否在修——一无所知，信息的不对等拉低了整体服务体验。

（4）**维修人员工作负荷缺乏实时可视**：管理者无法即时了解每名维修人员的在办任务量与完成效率，绩效评定缺少客观数据依据。

（5）**统计汇总过程低效**：学期末或年终时，人工筛查纸质台账或Excel表格来汇总维修数据，既慢又易出错，难以形成支持决策的量化依据。

近年来，Spring Boot与Vue等Web开发框架日趋成熟，移动互联网亦已广泛覆盖，搭建一套前后端分离的智慧校园报修系统在技术层面条件具备。借助B/S架构，师生可随时通过浏览器或手机提交报修并跟进进展，管理端则实现实时调度与统计，从而消解传统模式下信息割裂、流程黑箱的积弊[3]。

## 1.2 研究意义

本课题的研究意义分理论和应用两个层面。

**理论意义**方面，本研究围绕校园报修场景下的智能派单推荐展开探索，构建了一个融合任务负载、经验匹配度、区域熟悉度和历史评价四个维度的加权推荐模型，该模型具备可解释性，为智慧校园后勤调度方向的算法研究增添了实际案例。同时，本文针对JWT令牌与邮箱验证码两个场景，探讨了Redis缓存的降级容错策略，提出了一种轻量的高可用兜底方案。本系统基于SpringBoot+Vue前后端分离的工程实践，也可为同类高校信息化项目的技术选型与架构设计提供思路参考[4]。

**应用价值**方面，系统服务于学生、维修工、管理员三类群体，串联起报修提交、审核分派、维修作业到确认评价的完整业务链。WebSocket实时推送与SLA超时告警机制协同保障消息的即时触达与工单的时效管控。智能派单推荐辅助管理员做出更均衡的派单安排，提升维修人力资源的配置效率。统计可视化模块将工单数据映射为直观图表，为后勤管理部门的决策提供数据支撑。前端采用响应式设计，在PC宽屏与手机窄屏之间自动适配，契合当下大学生的使用场景，具备在高校部署运行的现实条件[5]。

## 1.3 国内外研究现状

**国外研究现状。** 欧美国家的高校信息化起步较早，以设施管理系统（Facility Management System）为代表的应用可追溯至上世纪90年代。美国高校普遍采用SchoolDude等商业运维平台，将工单提交、任务调度、资产管理及预防性维护整合到统一系统中[6]。近年来，物联网手段在校园设施管理中的渗透日益加深，借助建筑物内传感器网络实现故障自感知与预警[7]。日本高校在设施管理上偏重精细化与标准化，大多建立了规范化的运维流程体系和信息化管理平台[8]。国外方案的共同特征是以成熟商业软件为底座，功能覆盖面广、集成化程度高，但其管理范式与业务流程同中国高校后勤实际存在差异，直接引进往往面临适配成本高昂、本土化落地困难等障碍。

**国内研究现状。** 国内围绕高校后勤报修管理系统展开的研究在过去十年间逐步深化。刘锦[9]针对传统报修模式的缺陷，设计了一套以SpringBoot为基础的智慧校园故障报修信息管理系统，涵盖用户管理、工单管控、消息推送、评价反馈及数据统计等板块，并以ECharts实现可视化展示，该工作为本系统的功能架构提供了直接参照。山锋与丑洋[10]聚焦互联网新技术驱动的后勤报修系统，着重讨论了多端报修（Web+移动端）与信息化的必要性，指出数据统计对管理效率提升的推动力，其归纳的传统报修五大症结为本系统的需求分析奠定了理论根基。肖智兵等[11]以湖北职业技术学院为案例，实现了基于微信小程序的高校后勤报修系统，按报修人、维修人员、管理员划分功能模块，同时指出小程序虽然在便捷性上占优，但因绑定微信生态，在功能扩展和灵活性方面存在先天制约。

除报修系统之外，以SpringBoot+Vue为技术栈的管理系统在高校其他领域已有广泛落地，涉及学生管理、实验室管理、图书管理等场景的同类案例十分丰富[12][13][14]，充分验证了该前后端分离架构在高校信息化建设中的可行性和技术成熟度。

综合来看，已有工作在三个方向仍值得进一步改进：（1）工单分派多依赖人工指定或自由抢单，缺少数据驱动的智能推荐来辅助决策；（2）在系统容错层面，对Redis等中间件依赖的降级应对方案探讨不足；（3）大部分系统未引入SLA超时告警，工单堆积问题难以及时暴露和处置。本系统正是围绕上述短板，分别设计了智能派单推荐算法、Redis缓存降级策略以及SLA超时预警机制，希望为校园报修管理向智能化演进提供可参考的思路。

## 1.4 主要工作及论文组织结构

本论文围绕校园报修管理系统的设计与实现这条主线，依次从需求分析、技术选型、系统设计、功能开发和系统测试五个阶段推进，主要完成以下工作：

（1）对高校后勤报修的管理现状与痛点进行调研，梳理学生、维修工、管理员三类用户的功能与非功能需求，厘清从报修发起到服务评价的端到端业务流程。

（2）选定Spring Boot 2.7与Vue 3作为核心框架，采用前后端解耦的B/S架构，完成总体架构设计与技术方案论证。

（3）设计数据库模型（涵盖用户、工单、评价、日志、反馈及沟通消息六张核心表），搭建RESTful API接口体系，并实现基于Spring Security+JWT的认证鉴权层。

（4）完成核心功能模块的编码实现：用户认证（JWT+邮箱验证码）、工单状态机（含10种状态转换）、智能派单推荐（四维度加权评分）、WebSocket实时推送、SLA超时告警以及数据统计可视化（ECharts渲染）。

（5）编制测试用例，开展功能验证与兼容性测试，确保系统达到上线交付水平。

论文共分为七章，组织结构如下：

第1章 绪论。阐述系统开发背景与研究意义，梳理国内外相关工作的进展情况，说明本文主要工作与论文组织方式，提炼系统特色与创新点。

第2章 相关技术介绍。逐一说明系统开发所依赖的关键技术，涵盖MySQL 8.0、Redis 6.2、Spring Boot 2.7、MyBatis-Plus 3.5、Spring Security、JWT、Spring Mail、Vue 3、Element Plus、ECharts 6.0及WebSocket STOMP协议。

第3章 系统需求分析。由可行性评估、总体架构规划、功能需求拆解、业务流程建模、角色用例描述及非功能性约束六个维度展开需求分析。

第4章 系统设计。阐述总体架构、七大功能模块划分、数据库设计（包含E-R图与核心表结构）及接口设计（RESTful API规范与安全权限策略）。

第5章 系统实现。详细说明用户认证、工单状态机、智能派单推荐、WebSocket实时通知与SLA超时告警以及数据统计可视化五大核心模块的开发实现。

第6章 系统测试。介绍测试环境与测试目标，列出11项功能用例及其实验结果，分析测试阶段暴露的缺陷并给出修复策略。

第7章 总结与展望。归纳论文取得的主要成果，剖析系统当前存在的不足，展望后续可推进的优化方向。

## 1.5 系统特色与创新之处

本系统的特色与创新集中体现在以下四点：

（1）**智能派单推荐算法**：构建了融合任务负载、同类经验、区域经验与历史评分四个维度的加权推荐模型，具备可解释性，为管理员派单提供数据驱动的决策参考（详见5.3节）。

（2）**SLA超时预警机制**：基于定时调度实现超时工单的自动探测与告警推送，防止工单长期滞留无人处置（详见5.4节）。

（3）**Redis缓存降级策略**：在JWT令牌管理与邮箱验证码两个关键场景实现Redis至进程内存的自动回退，确保认证链路在缓存不可用时不中断（详见5.1节）。

（4）**移动端优先的响应式设计**：前端采用响应式布局方案，在手机窄屏与桌面宽屏之间自适应切换，确保高频操作在移动端即可流畅完成。
