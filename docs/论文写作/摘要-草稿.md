# 摘要

高校后勤报修是校园日常运行的支撑性环节，其数字化水平与智能化程度直接影响师生的服务感受和管理侧的运转效率。目前不少高校的报修处理仍停留在电话通知、纸质登记或Excel表格记录等方式，其短板集中体现在：上报途径狭窄、过程信息不透明、进度不可追溯、维修人员工作负载无法实时衡量、期末汇总统计耗时费力。针对上述痛点，本文基于SpringBoot与Vue的前后端分离架构，构建了一套校园报修管理系统。

系统将使用者划分为学生、维修工与管理员三类角色，打通了从故障提交、工单分配、维修执行直至服务评价的全链路闭环。服务端以Spring Boot 2.7为骨架，数据持久层采用MyBatis-Plus；Spring Security配合JWT令牌共同完成认证与权限校验，Redis 6.2承担Token缓存和邮箱验证码的暂存任务。客户端以Vue 3组合式API为核心，搭配Element Plus组件库构造自适应界面，ECharts 6.0驱动数据可视化呈现。派单环节引入了一套智能推荐机制，从维修人员的任务饱和度、同类故障处理经历、区域熟悉程度及历史评价得分四个维度综合计算，产出可解释的推荐分值，辅助管理员做出派单决策。SLA超时预警借助定时任务自动检测积压工单，并即时推送告警通知。Redis节点不可达时，系统会将缓存自动回退至进程级内存，确保认证链路持续可用。系统经11项功能用例验证，覆盖认证鉴权、工单流转、派单推荐、消息通知、统计图表及并发承载六个方向，全部用例通过，满足上线交付条件。

**关键词**：校园报修；Spring Boot；Vue；前后端分离；智能派单；工单状态机

# Abstract

University facility repair services form a critical part of campus operations, where digitization can produce measurable gains in service responsiveness and administrative efficiency. Conventional repair processes at many institutions still rely on phone calls, paper forms, or spreadsheet tracking, resulting in a narrow submission channel, opaque progress visibility, absence of real-time workload monitoring for technicians, and laborious manual data compilation. This thesis describes the design and implementation of a campus repair management system based on a SpringBoot+Vue front-end/back-end separated architecture.

Supporting three role categories — students, technicians, and administrators — the system encompasses the complete repair lifecycle from initial submission through final evaluation. On the server side, Spring Boot 2.7 serves as the core framework with MyBatis-Plus handling persistence. Authentication and access control are managed by Spring Security paired with JWT tokens, while Redis 6.2 provides token caching and email verification code storage. The client is built on Vue 3's Composition API with Element Plus delivering a responsive interface, and ECharts 6.0 renders data visualizations. A dispatch recommendation algorithm scores each technician across four weighted dimensions — current task load, relevant fault resolution experience, geographic area familiarity, and historical service ratings — producing an interpretable recommendation for administrator review. A scheduled SLA monitoring job periodically scans for overdue work orders and issues alerts. Should Redis become unreachable, the system degrades token and verification-code storage to in-process memory, preserving authentication continuity. The system was validated through 11 functional test cases spanning authentication, work order processing, dispatch recommendation, notification delivery, data statistics, and concurrent load, achieving a 100% pass rate.

**Keywords**: Campus Repair Management; Spring Boot; Vue; Front-end and Back-end Separation; Intelligent Dispatch; Work Order State Machine
