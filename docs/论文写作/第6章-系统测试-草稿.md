# 第6章 系统测试

本章从测试环境、测试目标、功能测试用例和测试结果四个方面说明系统测试过程。

本次测试使用的环境与开发阶段保持一致，各后端服务通过Docker Compose在本地进行容器化编排运行。相关的硬件参数及软件版本见表6.1。

## 6.2 测试目标

本次系统测试围绕以下五个方向展开：

（1）**功能完整性**：逐项验证七大功能模块中的所有功能点是否均可按照需求规格说明书的规定正确执行。

（2）**业务逻辑正确性**：重点检验工单状态机流转的合法性约束、智能派单推荐算法的评分准确性，以及三角色权限隔离的有效性。

（3）**数据一致性**：在高并发创建工单的场景下，验证是否存在数据丢失或主键冲突，并确认乐观锁机制能否正确感知并拦截并发修改冲突。

（4）**实时通信可靠性**：验证WebSocket STOMP长连接的稳定性、通知消息推送的准确送达，以及SLA超时预警功能是否按预期触发。

（5）**兼容性**：确认页面在Chrome、Edge、Firefox等多种浏览器以及移动端不同屏幕尺寸下均可正常渲染和交互[2]。

### 6.4.2 缺陷分析与修复

在测试推进过程中识别出的若干典型缺陷及其修复记录如下：

（1）**JWT令牌Redis认证失败**：测试初期，Redis 6.2实例开启了密码认证（requirepass），而Spring Boot 2.7配置中沿用了已废弃的`spring.data.redis.password`配置前缀。由于该前缀在Spring Boot 2.7版本中不再生效，导致Redis连接建立时密码实际未被传入，引发NOAUTH认证错误。采取的修复措施为：将Redis相关配置全部迁移至`spring.redis.password`（Spring Boot 2.7规范前缀），同时将Redis客户端实现由Lettuce更换为Jedis，从而彻底消除Redis 6.x的认证兼容性障碍。

（2）**LocalDateTime序列化异常**：前端提交的评价数据中携带LocalDateTime类型字段，而Jackson默认序列化器并未内置对Java 8日期时间类型的处理能力，序列化过程中触发异常。修复方案为：向Jackson的ObjectMapper配置中注册`JavaTimeModule`模块，使LocalDateTime类型被规范地序列化为ISO-8601格式字符串。

（3）**雪花ID前端精度丢失**：后端采用雪花算法生成的64位Long型主键在传输至前端后，由于JavaScript的Number类型至多只能精确表达53位以内的整数，超出部分被截断为零，致使前后端ID不一致。修复方案为：在Jackson配置中启用`SerializationFeature.WRITE_NUMBERS_AS_STRINGS`特性，将Long型ID统一以字符串形式序列化传输，确保数值精度不受损失。

上述缺陷在测试阶段已全部修复，各相关测试用例的回归验证均已通过。

### 6.4.3 兼容性测试

系统前端页面在Chrome 120+、Edge 120+、Firefox 121+三款主流浏览器上完成了兼容性验证。各页面的布局排版、表单控件交互、图表渲染效果以及WebSocket连接状态在各浏览器中表现均保持一致，未观察到兼容性差异问题。移动端的响应式适配在375px（模拟iPhone）、414px（模拟Android主流机型）和768px（平板设备）三种视口宽度下均能正常呈现与操作，底部导航栏和触摸交互的响应均符合预期。
