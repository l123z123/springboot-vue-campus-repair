# 校园报修管理系统 - 核心业务需求文档（派单版）

## 1. 项目背景与目标
- **项目名称**: 基于 SpringBoot + Vue 的校园报修管理系统
- **核心模式**: **管理员审核派单 + 智能派单推荐**
- **目标用户**: 学生（报修）, 维修工（接收派单/维修）, 管理员（审核/派单/统计）
- **核心价值**: 解决报修流程不透明、人工派单效率低的问题，通过可解释的智能派单推荐、SLA 超时预警与实时通知提升后勤处理效率。

### 1.1 答辩版需求边界（冻结）
- **前端形态**：本项目是 Vue3 Web 应用，不按微信/支付宝小程序交付。学生端、维修工端采用“移动优先响应式 Web / H5”体验，并补强 PC 宽屏展示；管理员端为 PC 后台布局。
- **主流程优先**：学生提交报修 → 管理员审核 → 系统推荐维修工 → 管理员派单 → 维修工处理 → 学生确认与评价 → 管理端统计与通知。
- **不再扩展无关功能**：答辩版不新增支付、地图、复杂设备二维码、复杂排班等与报修派单主线弱相关的功能。
- **账号与认证**：学生可公开注册，维修工由管理员创建；支持 JWT 登录、邮箱验证码注册/登录，邮箱发码可配 QQ SMTP。
- **验收口径**：以本文件、`RepairOrderStatus`、`backend/src/main/resources/schema.sql` 和 `QA_回归验收清单.md` 为最终演示依据。

## 2. 核心业务流程 (State Machine)
**实现以代码为准**：`com.campus.repair.enums.RepairOrderStatus` 与数据库 `repair_order.status` 使用同一套数字码。主流程（派单版，含审核与多中间态）如下表；合法后继以枚举内 `canTransitionTo` 为准：

| 码 | 含义 | 行为要点 |
|---|------|----------|
| 0 | 待审核 | 学生提交报修 |
| 1 | 审核中 | 管理员处理 |
| 3 | **待派单** | 管理员审核通过后等待派单 |
| 4 | 已派单 | 已绑定维修工，等待维修工开始处理 |
| 5 | 维修中 | 上门/处理 |
| 6 | 维修完成 | 维修工完工 |
| 7 | 学生确认 | 学生确认结果 |
| 8 | 已评价 | 流程结束态之一 |
| 9 / 10 | 已拒绝 / 已取消 | 终态分支 |

**文字版主线**：提交(0) → 审核(1) → 待派单(3) 或直接已派单(4) → 维修中(5) → 维修完成(6) → 学生确认(7) → 已评价(8)。审核拒绝(9)、取消(10) 等见枚举合法流转。

## 3. 关键功能模块关键词

### 3.1 学生端
- **报修提交**: 表单验证、多图上传、位置选择。
- **进度追踪**: 实时查看当前状态（审核中/待派单/已派单/维修中/已完成）。
- **消息通知**: 接收状态变更通知（如：管理员已派单、维修工已完工）。
- **评价体系**: 完工后对维修工打分 (1-5星) 及留言。

### 3.2 维修工端
- **我的派单任务**:
  - 展示管理员派给自己的工单。
  - 维修工确认开始维修后，状态进入“维修中”。
- **工单处理**: 更新状态 (开始维修 -> 完工)，上传维修后照片。
- **历史记录**: 查看已完成的工单及收入统计。

### 3.3 管理端
- **用户管理**: 添加/禁用维修工账号 (维修工不由学生注册，由管理员创建)。
- **订单审核**: 审核学生提交的原始工单。
- **智能派单推荐**: 审核或派单时，根据维修工负载、同类经验、同区域经验和历史评分推荐合适维修工，管理员可采用或手动调整。
- **SLA 超时监控**: 若工单长时间处于待审核、待派单、已派单、维修中等未闭环状态，触发管理员预警（见 `repair.sla.pending-hours`）。
- **数据大屏**: 统计完成率、平均响应时间、故障类型分布、维修工绩效等。

## 4. 非功能性需求 (NFR)
- **智能派单可解释**: 推荐结果必须给出原因，如当前在办数量、同类处理经验、同区域经验、历史评分等，避免黑盒决策。
- **数据一致性**: 状态流转必须原子化，避免状态错乱。
- **实时性**: 状态变更后，前端需尽快感知 (通过轮询或消息推送)。
- **本地化**: 所有数据和服务必须在本地运行，不依赖云端。
