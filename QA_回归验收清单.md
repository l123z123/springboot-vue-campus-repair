# 校园报修系统回归验收清单

## 一、环境健康检查

- Docker 容器：`mysql`、`redis`、`rabbitmq` 均为运行状态。
- 后端健康：`GET /api/actuator/health` 返回 `UP`。
- 前端启动后，`/api/auth/login` 可正常返回 `code=200`。
- 旧库升级时已执行 `sql/migration_add_sys_user_email.sql`，`sys_user` 至少包含 `email`、`signature`、`gender`、`avatar_url` 字段。
- 前端 PC 宽度（建议 1366px）下：学生端/维修工端显示侧栏与桌面化卡片；手机宽度下仍显示底部导航。

## 二、学生主流程

1. 学生注册：填写用户名、邮箱、邮箱验证码、密码、姓名、手机号、院系；验证码可通过 QQ SMTP 或后端日志/内存降级完成。
2. 学生可通过密码登录；已绑定邮箱的账号可通过邮箱验证码登录。
3. 学生提交普通报修（非急修），至少填写描述或图片、校区、区域、类型、手机。
4. 学生在详情页可看到：提交时间、状态为待审核。
5. 管理员审核通过后，学生可看到待派单或已派单（按审核选项）。
6. 维修完成后，学生可确认完成并提交评价。
7. 评价后状态到已评价，时间线顺序正确且无伪造时间。

## 三、管理员流程

1. 审核拒绝：状态转为已拒绝，学生端可见拒绝结果。
2. 审核通过-暂不派单：状态变为待派单，`repairmanId` 为空。
3. 审核通过-指定维修工：状态变为已派单，写入 `repairmanId` 与 `dispatchTime`。
4. 智能派单推荐可展示推荐维修工、推荐分和推荐理由，点击“采用”能填入维修工。
5. 指派维修工时，无法选择停用账号、非维修工账号、报修人本人。

## 四、维修工流程

1. 维修工只可见管理员派给自己的工单。
2. 维修工点击开始维修后，状态变为维修中，开始时间写入，学生收到状态通知。
3. 未被派单的维修工无法操作他人工单。
4. 完成维修后状态变为维修完成，完工时间写入。
5. 非本人维修工无法操作他人已派工单。

## 五、消息与通知

1. 系统通知中“状态更新”点击应跳工单详情，不跳聊天页。
2. 聊天类通知点击跳到工单沟通页。
3. 底部消息 Tab 有未读红点/数字，会话行有未读数。
4. 进入某工单聊天后，该工单未读数清零并调用后端已读接口。
5. 管理员收到“SLA超时预警”通知时，铃铛中展示涉及工单号；点击工单号可进入对应详情。

## 六、稳定性与异常提示

- MySQL 不可用时，接口应返回明确的 `503` 数据库不可用提示。
- Redis 不可用时，登录仍可走 JWT 降级；邮箱验证码可使用本进程内存降级，不应直接 503。
- 前端 GET 请求遇到短暂网络抖动可自动重试一次，不影响主流程。
- 401 仍会正确清理登录态并跳转登录页。

## 七、发布前最小检查命令

- 后端：`mvn -q compile -DskipTests`
- 前端：`npm run build`
- 手工冒烟：按本清单二到五条至少走一遍。
